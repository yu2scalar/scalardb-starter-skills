# scalardb-generate-springboot-starter

> Status: **v0.1.0 (2026-07-14 ‚Äî plan-006 P6 initial implementation)**
> Targets ScalarDB Cluster **3.18.0**, Spring Boot **3.5.13**, Java **17**, Gradle (wrapper bundled).
> Skill 6 of 8 in the scalardb-starter-skills walk-through.

## Overview

Reads a Schema Loader JSON and generates, for **every table**, a complete REST CRUD stack in two flavors side by side:

| flavor | REST base path | classes per table |
|---|---|---|
| Java (CRUD) API | `/<table-kebab>` | `<Table>Controller` / `<Table>Service` / `<Table>Repository` |
| SQL API | `/<table-kebab>-sql` | `<Table>SqlController` / `<Table>SqlService` / `<Table>SqlRepository` |

plus shared classes per table (`<Table>` model, `<Table>Dto`, `<Table>Mapper`) and one-time infrastructure (configs, retry templates, `ApiResponse`, `CustomException`).

The same table is reachable through both flavors ‚Äî call `/orders` and `/orders-sql` with identical payloads to compare the two APIs. This mirrors the reference implementations the patterns were taken from (see `references/patterns/README.md`).

### Endpoints generated per table

| method + path | operation |
|---|---|
| `POST /<table>` | insert |
| `POST /<table>/upsert` | upsert |
| `GET /<table>/{pk...}/{ck...}` | get by full key (all key parts as ordered path variables) |
| `PUT /<table>` | update (body) |
| `DELETE /<table>/{pk...}/{ck...}` | delete by full key |
| `GET /<table>/scan-by-pk/{pk...}` | list within one partition (only for tables **with** a clustering key) |
| `GET /<table>/scan-all` | full scan ‚Äî generated **only** when the table's storage has `cross_partition_scan.enabled=true` in the marker (`config.storages`); otherwise omitted (it would fail on NoSQL and is disabled by default) |

The same set is emitted at `/<table>-sql` for the SQL flavor.

### What this skill does NOT do

- Choose only one API flavor (both are always generated ‚Äî delete one package if unwanted)
- Generate tests (ask Claude Code afterwards)
- BFF / two-phase-commit patterns (future material)
- Start the cluster or load the schema

## Phase 0 ‚Äî project root and inputs

1. Project root: marker convention as in `scalardb-generate-config` (existing `./.scalardb-starter-skills.json` ‚Üí cwd; otherwise ask a project name and create the directory + marker). The Spring Boot project is generated **directly in the project root** (`build.gradle` next to `scalardb/`).
2. Schema file:

   ```
   Q1. Schema file? (default: scalardb/schema/schema.json when present;
       otherwise enter a path to any Schema Loader JSON)
   ```

   Parse it: table full names (`namespace.table`), `partition-key`, `clustering-key` (with ASC/DESC), `columns` (name ‚Üí ScalarDB type). Reject non-`transaction: true` tables with a warning (starter targets transactional tables).
3. Application name / package:

   ```
   Q2. Application name? (default: project directory name)
   Q3. Base package? (default: com.example.<app_name_snake>)
   ```

4. Show the table list and the endpoint plan (including which tables get `scan-all`) and confirm.

## Phase 1 ‚Äî fixed scaffolding

- Render `templates/build.gradle.tmpl` (`GROUP` = package minus last segment, `SCALARDB_VERSION` = 3.18.0), `settings.gradle.tmpl`, `application.properties.tmpl`, `Application.java.tmpl`.
- Copy the Gradle wrapper (`templates/gradlew`, `gradlew.bat`, `gradle/wrapper/*`) and `chmod +x gradlew`.
- Generate the one-time classes into the base package, adapted **verbatim in behavior** from `references/patterns/` (replace only the package): `config/ScalarDbConfig`, `config/ScalarDbSqlConfig`, `config/RetryProperties`, `config/CoreApiRetryConfig`, `config/SqlApiRetryConfig`, `dto/ApiResponse`, `dto/ResponseStatusDto`, `exception/CustomException`, and `repository/SqlRepository` (SQL base class).
- `application.properties` points `scalardb.config.file` / `scalardb_sql.config.file` at `scalardb/config/‚Ä¶` ‚Äî the client properties stay the single source of truth; never copy them into `src/main/resources`.

## Phase 2 ‚Äî per-table generation

For each table, generate the class set following `references/patterns/` (the `ProductTestAllTypesTest` example covers every data type and a PK+CK table ‚Äî adapt names, keys, and types; keep the transaction/retry/rollback structure exactly):

- `model/<Table>.java` ‚Äî Lombok builder model with `NAMESPACE` / `TABLE` / column-name constants, `getPartitionKey()` / `getClusteringKey()` (omit the clustering-key method for PK-only tables).
- `dto/<Table>Dto.java`, `mapper/<Table>Mapper.java`.
- Java API: `repository/<Table>Repository.java` (Get / Insert / Update+`updateIfExists` / Upsert / Delete+`deleteIfExists` / Scan-by-PK; Scan-all only when applicable), `service/<Table>Service.java` (RetryTemplate around `manager.start()` ‚Ä¶ `commit()`, rollback on failure, `determineErrorCode`), `controller/<Table>Controller.java` (`@RequestMapping("/<table-kebab>")`, `ApiResponse` wrapper).
- SQL API: `repository/<Table>SqlRepository.java` (extends `SqlRepository`, prepared statements), `service/<Table>SqlService.java`, `controller/<Table>SqlController.java` (`@RequestMapping("/<table-kebab>-sql")`).

Type mapping (ScalarDB ‚Üí Java): TEXT‚Üí`String`, INT‚Üí`Integer`, BIGINT‚Üí`Long`, FLOAT‚Üí`Float`, DOUBLE‚Üí`Double`, BOOLEAN‚Üí`Boolean`, BLOB‚Üí`byte[]`, DATE‚Üí`LocalDate`, TIME‚Üí`LocalTime`, TIMESTAMP‚Üí`LocalDateTime`, TIMESTAMPTZ‚Üí`Instant`.

Key path variables appear in key-declaration order: partition keys first, then clustering keys (e.g. `GET /order-items/{order_id}/{item_id}`).

## Phase 3 ‚Äî compile check

Offer to run:

```
./gradlew compileJava --no-daemon
```

(first run downloads dependencies ‚Äî needs network and a few minutes). Fix any generation errors before reporting success; report honestly if compilation was skipped.

## Phase 4 ‚Äî report

- Generated tree summary + endpoint table.
- Run instructions:
  1. ensure the cluster is reachable (ClusterIP: `kubectl port-forward svc/<release>-envoy 60053:60053 -n <namespace>` from the marker; LoadBalancer: IP already in the properties),
  2. `./gradlew bootRun`,
  3. Swagger UI: `http://localhost:8080/swagger-ui/index.html`, plus one `curl` example per flavor.
- Marker merge: `{ "app": { "package": "...", "tables": [...], "generatedAt": "<ISO-8601>" } }`
- Next: `/scalardb-validate-config` to audit the configuration, `/scalardb-stop-sample-db` when finished with the demo databases.
