# scalardb-generate-springboot-starter

> Status: **v0.2.1 (2026-07-23 — plan-007: pre-run placeholder guard + DB-CLUSTER-30001 troubleshooting; v0.2.0 2026-07-17 = scenario endpoints for presets A/B; v0.1.0 2026-07-14 = plan-006 P6 initial implementation)**
> Targets ScalarDB Cluster **3.18.0**, Spring Boot **3.5.13**, Java **17**, Gradle (wrapper bundled).
> Skill 6 of 9 in the scalardb-starter-skills walk-through.

## Overview

Reads a Schema Loader JSON and generates, for **every table**, a complete REST CRUD stack in two flavors side by side:

| flavor | REST base path | classes per table |
|---|---|---|
| Java (CRUD) API | `/<table-kebab>` | `<Table>Controller` / `<Table>Service` / `<Table>Repository` |
| SQL API | `/<table-kebab>-sql` | `<Table>SqlController` / `<Table>SqlService` / `<Table>SqlRepository` |

plus shared classes per table (`<Table>` model, `<Table>Dto`, `<Table>Mapper`) and one-time infrastructure (configs, retry templates, `ApiResponse`, `CustomException`).

The same table is reachable through both flavors — call `/orders` and `/orders-sql` with identical payloads to compare the two APIs. This mirrors the reference implementations the patterns were taken from (see `references/patterns/README.md`).

### Endpoints generated per table

| method + path | operation |
|---|---|
| `POST /<table>` | insert |
| `POST /<table>/upsert` | upsert |
| `GET /<table>/{pk...}/{ck...}` | get by full key (all key parts as ordered path variables) |
| `PUT /<table>` | update (body) |
| `DELETE /<table>/{pk...}/{ck...}` | delete by full key |
| `GET /<table>/scan-by-pk/{pk...}` | list within one partition (only for tables **with** a clustering key) |
| `GET /<table>/scan-all` | full scan — generated **only** when the table's storage has `cross_partition_scan.enabled=true` in the marker (`config.storages`); otherwise omitted (it would fail on NoSQL and is disabled by default) |

The same set is emitted at `/<table>-sql` for the SQL flavor.

### Scenario endpoints (presets A / B only)

When the schema matches a preset (see Phase 2.5), the starter additionally gets **one multi-table business endpoint per flavor** — the walk-through's highlight, because with the recommended demo mapping (`order`/`account` on PostgreSQL, `inventory`/`audit` on MySQL) a single request runs **one ACID transaction spanning two real databases**:

| preset | endpoints | one transaction does |
|---|---|---|
| A | `POST /orders/place`, `POST /orders-sql/place` | read `inventory.items` stock → abort if missing/insufficient → insert `order.orders` + `order.order_items` → decrement stock → commit |
| B | `POST /accounts/transfer`, `POST /accounts-sql/transfer` | read both `account.accounts` → abort if missing/insufficient → update both balances → insert `audit.transfer_history` (one row per side) → commit |

Custom (non-preset) schemas get CRUD only — say so in the report.

### What this skill does NOT do

- Choose only one API flavor (both are always generated — delete one package if unwanted)
- Generate tests (ask Claude Code afterwards)
- BFF / two-phase-commit patterns (future material)
- Start the cluster or load the schema

## Phase 0 — project root and inputs

1. Project root: marker convention as in `scalardb-generate-config` (existing `./.scalardb-starter-skills.json` → cwd; otherwise ask a project name and create the directory + marker). The Spring Boot project is generated **directly in the project root** (`build.gradle` next to `scalardb/`).
2. Schema file:

   ```
   Q1. Schema file? (default: scalardb/schema/schema.json when present;
       otherwise enter a path to any Schema Loader JSON)
   ```

   Parse it: table full names (`namespace.table`), `partition-key`, `clustering-key` (with ASC/DESC), `columns` (name → ScalarDB type). Reject non-`transaction: true` tables with a warning (starter targets transactional tables).
3. Application name / package:

   ```
   Q2. Application name? (default: project directory name)
   Q3. Base package? (default: com.example.<app_name_snake>)
   ```

4. Show the table list and the endpoint plan (including which tables get `scan-all`, and whether scenario endpoints will be generated — Phase 2.5 preset match) and confirm.

## Phase 1 — fixed scaffolding

- Render `templates/build.gradle.tmpl` (`GROUP` = package minus last segment, `SCALARDB_VERSION` = 3.18.0), `settings.gradle.tmpl`, `application.properties.tmpl`, `Application.java.tmpl`.
- Copy the Gradle wrapper (`templates/gradlew`, `gradlew.bat`, `gradle/wrapper/*`) and `chmod +x gradlew`.
- Generate the one-time classes into the base package, adapted **verbatim in behavior** from `references/patterns/` (replace only the package): `config/ScalarDbConfig`, `config/ScalarDbSqlConfig`, `config/RetryProperties`, `config/CoreApiRetryConfig`, `config/SqlApiRetryConfig`, `dto/ApiResponse`, `dto/ResponseStatusDto`, `exception/CustomException`, and `repository/SqlRepository` (SQL base class).
- `application.properties` points `scalardb.config.file` / `scalardb_sql.config.file` at `scalardb/config/…` — the client properties stay the single source of truth; never copy them into `src/main/resources`.

## Phase 2 — per-table generation

For each table, generate the class set following `references/patterns/` (the `ProductTestAllTypesTest` example covers every data type and a PK+CK table — adapt names, keys, and types; keep the transaction/retry/rollback structure exactly):

- `model/<Table>.java` — Lombok builder model with `NAMESPACE` / `TABLE` / column-name constants, `getPartitionKey()` / `getClusteringKey()` (omit the clustering-key method for PK-only tables).
- `dto/<Table>Dto.java`, `mapper/<Table>Mapper.java`.
- Java API: `repository/<Table>Repository.java` (Get / Insert / Update+`updateIfExists` / Upsert / Delete+`deleteIfExists` / Scan-by-PK; Scan-all only when applicable), `service/<Table>Service.java` (RetryTemplate around `manager.start()` … `commit()`, rollback on failure, `determineErrorCode`), `controller/<Table>Controller.java` (`@RequestMapping("/<table-kebab>")`, `ApiResponse` wrapper).
- SQL API: `repository/<Table>SqlRepository.java` (extends `SqlRepository`, prepared statements), `service/<Table>SqlService.java`, `controller/<Table>SqlController.java` (`@RequestMapping("/<table-kebab>-sql")`).

Type mapping (ScalarDB → Java): TEXT→`String`, INT→`Integer`, BIGINT→`Long`, FLOAT→`Float`, DOUBLE→`Double`, BOOLEAN→`Boolean`, BLOB→`byte[]`, DATE→`LocalDate`, TIME→`LocalTime`, TIMESTAMP→`LocalDateTime`, TIMESTAMPTZ→`Instant`.

Key path variables appear in key-declaration order: partition keys first, then clustering keys (e.g. `GET /order-items/{order_id}/{item_id}`).

## Phase 2.5 — scenario endpoints (presets only)

**Detection**: the marker's `schema.preset` (written by `scalardb-generate-schema-file`) is `"A"` or `"B"`, **or** the parsed schema structurally contains the preset's tables with the columns the scenario touches (A: `order.orders`, `order.order_items`, `inventory.items` with `stock` + `unit_price`; B: `account.accounts` with `balance`, `audit.transfer_history`). No match → skip this phase.

Generate both flavors with the same transaction/retry/rollback structure as Phase 2 services (RetryTemplate, rollback on failure, `determineErrorCode`); business aborts call `transaction.abort()` / `ROLLBACK` and map to **HTTP 409** with a scenario error code in the `ApiResponse` wrapper. Shared DTOs live in `dto/`; scenario services/controllers are separate classes so the CRUD ones stay untouched.

### Preset A — place order

- Classes: `dto/PlaceOrderRequest` / `dto/PlaceOrderResponse`, `service/OrderScenarioService` (Java API), `service/OrderScenarioSqlService` (SQL API), `controller/OrderScenarioController` (`@RequestMapping("/orders")`, `@PostMapping("/place")`), `controller/OrderScenarioSqlController` (`/orders-sql/place`).
- Request: `{ "customerId": "c1", "items": [ { "itemId": "i1", "quantity": 2 } ] }` — validate: `items` non-empty, `quantity >= 1`, no duplicate `itemId` (else HTTP 400 before any transaction).
- In **one transaction**:
  1. For each item: Get `inventory.items[item_id]`. Missing → abort, 409 `ITEM_NOT_FOUND` (include `itemId`). `stock < quantity` → abort, 409 `INSUFFICIENT_STOCK` (include `itemId`, `stock`, `requested`).
  2. `order_id` = random UUID, `created_at` = epoch millis, `total_amount` = Σ `quantity × items.unit_price` (price read in this transaction — the price at purchase time).
  3. Insert `order.orders` (`status` = `"PLACED"`); insert one `order.order_items` row per item (`unit_price` copied from `inventory.items`).
  4. Update each `inventory.items.stock` to `stock − quantity`.
  5. Commit.
- Response (`ApiResponse`-wrapped): `{ "orderId", "customerId", "status", "totalAmount", "createdAt", "items": [ { "itemId", "quantity", "unitPrice" } ] }`.

### Preset B — transfer

- Classes: `dto/TransferRequest` / `dto/TransferResponse`, `service/AccountScenarioService` / `AccountScenarioSqlService`, `controller/AccountScenarioController` (`/accounts/transfer`), `controller/AccountScenarioSqlController` (`/accounts-sql/transfer`).
- Request: `{ "fromAccountId": "a1", "toAccountId": "a2", "amount": 100 }` — validate: `amount >= 1`, `fromAccountId != toAccountId` (else HTTP 400).
- In **one transaction**:
  1. Get both `account.accounts` rows. Missing → abort, 409 `ACCOUNT_NOT_FOUND` (say which). `from.balance < amount` → abort, 409 `INSUFFICIENT_BALANCE` (include `balance`, `requested`).
  2. `transfer_id` = random UUID, `created_at` = epoch millis. Update both balances (`from −amount`, `to +amount`) and `updated_at`.
  3. Insert **two** `audit.transfer_history` rows — one per side, same `transfer_id`: partition key = that side's `account_id`, `counterpart_account_id` = the other side, `amount` = `−amount` on the from side / `+amount` on the to side (so `GET /transfer-history/scan-by-pk/{account_id}` shows each account's history newest-first via the DESC clustering key).
  4. Commit.
- Response: `{ "transferId", "fromAccountId", "toAccountId", "amount", "fromBalance", "toBalance" }` (post-transfer balances).

## Phase 3 — compile check

Offer to run:

```
./gradlew compileJava --no-daemon
```

(first run downloads dependencies — needs network and a few minutes). Fix any generation errors before reporting success; report honestly if compilation was skipped.

## Phase 4 — report

- Generated tree summary + endpoint table.
- **Pre-run guard**: `grep -n 'ENVOY_LOAD_BALANCER_IP' scalardb/config/*.properties`. If the placeholder is still there, do **not** tell the user to run yet — the app would start and fail on the first call. Point them to `/scalardb-start-scalardb-cluster` to resolve and confirm the reachable contact point first (it writes it into both properties files).
- Run instructions:
  1. ensure the cluster is reachable (ClusterIP: `kubectl port-forward svc/<release>-envoy 60053:60053 -n <namespace>` from the marker; LoadBalancer: IP already in the properties),
  2. `./gradlew bootRun`,
  3. Swagger UI: `http://localhost:8080/swagger-ui/index.html`, plus one `curl` example per flavor — and, when scenario endpoints were generated, the scenario `curl` first (it is the walk-through's payoff).
- Troubleshooting pointers for the first call (the app cannot reach the cluster):
  - `DB-CLUSTER-30001: ... Cluster Node IP Address: <ENVOY_LOAD_BALANCER_IP>` — `contact_points` is still the **unreplaced placeholder**. Resolve it via `/scalardb-start-scalardb-cluster` (Phase 5) or set a reachable IP/hostname in **both** `scalardb/config/*.properties`, then restart the app.
  - `UNAVAILABLE: io exception` from any endpoint — `contact_points` holds a value but it is not reachable from where the app runs: LoadBalancer IP pending because `minikube tunnel` is not running? `indirect:localhost` but the port-forward is not active (or bound to a different `--address`) on the machine running the app?
- Marker merge: `{ "app": { "package": "...", "tables": [...], "scenario": "A" | "B" | null, "generatedAt": "<ISO-8601>" } }`
- Next: `/scalardb-validate-config` to audit the configuration, `/scalardb-stop-sample-db` when finished with the demo databases.
