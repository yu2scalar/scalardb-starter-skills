# scalardb-generate-schema-file

> Status: **v0.1.0 (2026-07-14 — plan-006 P5 initial implementation)**
> Targets ScalarDB Cluster **3.18.0** (loader image `ghcr.io/scalar-labs/scalardb-cluster-schema-loader:3.18.0`).
> Skill 5 of 9 in the scalardb-starter-skills walk-through.

## Overview

Writes `scalardb/schema/schema.json` (from one of two presets) and `scalardb/schema/load-schema.sh`, which loads the schema through the running cluster by launching the Cluster Schema Loader as a **Kubernetes Job** — no local Java or Docker needed, and no port-forward network issues.

### What this skill does NOT do

- Design arbitrary schemas interactively (pick a preset, then edit `schema.json` by hand if needed — the format is plain Schema Loader JSON; `scalardb-generate-springboot-starter` also accepts any schema JSON you provide)
- Start the cluster (`scalardb-start-scalardb-cluster`) or generate config
- Data-type / partition-design consulting (see `references/20-data-model-and-types.md`)

## Phase 0 — project root

Same convention as `scalardb-generate-config`: if `./.scalardb-starter-skills.json` exists, the current directory is the project root; otherwise ask for a project name and create `<project-name>/` + marker. Read `config.k8sNamespace`, `config.releaseName`, `config.authEnabled`, and `config.namespaces` when present (standalone use without them is fine — ask for namespace/release then).

## Phase 1 — preset selection

```
Q1. Which schema preset?
    1) A: Order-Inventory (recommended) — order.orders, order.order_items,
       inventory.items — includes a clustering-key example (order_items)
    2) B: Transfer — account.accounts, audit.transfer_history — includes a
       DESC clustering-key example (transfer_history)
```

Preset definitions (all tables `transaction: true`):

| preset | table | partition key | clustering key | other columns |
|---|---|---|---|---|
| A | `order.orders` | `order_id` TEXT | — | `customer_id` TEXT, `status` TEXT, `total_amount` INT, `created_at` BIGINT |
| A | `order.order_items` | `order_id` TEXT | `item_id` ASC | `quantity` INT, `unit_price` INT |
| A | `inventory.items` | `item_id` TEXT | — | `name` TEXT, `stock` INT, `unit_price` INT |
| B | `account.accounts` | `account_id` TEXT | — | `owner_name` TEXT, `balance` INT, `updated_at` BIGINT |
| B | `audit.transfer_history` | `account_id` TEXT | `created_at` DESC | `transfer_id` TEXT, `counterpart_account_id` TEXT, `amount` INT |

**Namespace cross-check**: if the marker has `config.namespaces`, compare with the preset's namespaces. On mismatch (e.g. config mapped `order`/`inventory` but preset B is chosen), warn: unmapped namespaces fall back to `scalar.db.multi_storage.default_storage` — offer to (a) continue anyway (works, all tables land on the default storage), or (b) switch the preset, or (c) stop and re-run `/scalardb-generate-config` to fix the mapping. Never edit `custom-values.yaml` silently.

## Phase 2 — write files

```
scalardb/schema/
├── schema.json      (copy of templates/schema-preset-<a|b>.json)
└── load-schema.sh   (rendered from templates/load-schema.sh.tmpl, chmod +x)
```

Template tokens: `K8S_NAMESPACE`, `RELEASE_NAME`, `SCALARDB_VERSION` (=3.18.0), `AUTH_ENABLED`, `AUTH_USERNAME`, `AUTH_PASSWORD` (admin/admin when auth is on — noted in the README as a demo default to change).

`load-schema.sh` behavior: ConfigMap (schema.json + in-cluster loader properties pointing at `indirect:<release>-envoy.<namespace>.svc.cluster.local`) → one-shot Job with `--coordinator` (also creates the coordinator tables) → wait for completion → print loader logs → clean up (Job/ConfigMap are kept on failure for debugging). Re-running is safe; existing tables are left as they are.

Marker merge:

```json
{ "schema": { "preset": "A", "file": "scalardb/schema/schema.json", "generatedAt": "<ISO-8601>" } }
```

## Phase 3 — offer to load now

If the marker shows `cluster.status == "running"`, ask:

```
Q2. Load the schema into the running cluster now? (y/n)
```

On `y`, run `bash scalardb/schema/load-schema.sh` (apply the shared context-mismatch convention first) and merge `"loadedAt"` into the marker's `schema` section on success. On `n` (or no running cluster), print the manual command.

## Phase 4 — report

- File list, chosen preset, table summary.
- Reminder: to customize, edit `schema.json` and re-run `load-schema.sh` (new tables are created; existing ones are not altered — column changes need manual DDL planning).
- Next: `/scalardb-generate-springboot-starter` (generates CRUD REST APIs from this schema.json, or from any schema JSON you point it at).
